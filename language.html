Compiler Construction

Overview

A semester-long solo project building a compiler incrementally in Racket, targeting a86 — a simplified subset of x86-64 assembly represented as Racket data structures. Each assignment extended the compiler with new language features, from basic primitives all the way to pattern matching and exception handling.



Features Implemented

  • Primitives & Cond — added abs, unary -, not, and cond expressions with type-error signalling
  • Case expressions — dispatch on integer/boolean values, compiled to chains of cmp and je instructions
  • Multi-binding let and let* — generalized let to bind any number of variables simultaneously; let* with forward-referencing bindings
  • List & Vector primitives — heap-allocated values; implemented length, reverse, list->vector, vector->list with type checking
  • Variable-arity & case-lambda — rest arguments collected into lists at runtime; arity dispatch compiled to generated check-and-branch sequences
  • Pattern matching — compiled match expressions for list, vector, and predicate patterns against nested structures
  • Exception handling — implemented raise and with-handlers; used a designated register to hold a pointer to the current handler on the stack, with each handler storing predicate, function, return label, and parent handler pointer for nested handlers.

Every feature added was implemented at two levels: the interpreter (to define correct behavior) and the compiler (to generate correct a86 assembly).

Secure ATM & Bank System

Team (Build) · Solo (Break)

A team project to design and implement a secure ATM and banking system in C. The system had to remain secure even if the attacker knew a user's PIN but not their card, or had their card but not their PIN. It also had to account for packet sniffing or replay attacks.



My Contribution

I was responsible for the ATM client program which handled all user-facing interaction and constructed encrypted messages to the bank. This included session management, command parsing, and send messgaes to the bank program. I also implemented PIN hashing using OpenSSL's EVP message digest API: rather than storing raw PINs in card files, the system hashes each PIN with SHA-256 and a salt, so even a stolen card file reveals nothing about the actual PIN.



Vulnerabilities Defended Against

  • Replay attacks — per-message nonces and monotonically increasing sequence numbers inside the GCM-authenticated payload prevent the attacker from reusing captured packets.
  • Packet tampering — AES-256-GCM's authentication tag detects any in-transit modification; tampered messages are dropped by the bank.
  • Plaintext PIN storage — SHA-256 hashing with salt means card files never contain recoverable PIN data.
  • Buffer overflow — strict input length validation across all ATM commands prevents stack corruption and potential code injection.
  • Integer overflow on withdrawal — bounds checking on withdrawal amounts prevents negative-value exploits that would effectively deposit money into an account.


Break-it

In the offensive phase, I analyzed another team's implementation and identified that they had no rate limiting or lockout mechanism on failed PIN attempts. I wrote a C program that generated all 10,000 four-digit PIN combinations, formatted each as a valid begin-session command, and fed them sequentially into their ATM program. Since the team's ATM accepted unlimited incorrect attempts without tracking failures, the program successfully brute-forced the correct PIN and authenticated as the target user.
